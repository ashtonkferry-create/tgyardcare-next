Table 1: index_coverage_log
```sql
CREATE TABLE IF NOT EXISTS index_coverage_log (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    page_url TEXT NOT NULL,
    check_date DATE NOT NULL,
    verdict TEXT,              -- 'PASS', 'FAIL', 'NEUTRAL', 'VERDICT_UNSPECIFIED'
    coverage_state TEXT,       -- e.g., 'Submitted and indexed'
    indexing_state TEXT,       -- from IndexingState enum
    last_crawl_time TIMESTAMPTZ,
    page_fetch_state TEXT,
    robots_txt_state TEXT,
    google_canonical TEXT,
    previous_verdict TEXT,     -- for change detection
    alert_sent BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE UNIQUE INDEX idx_index_coverage_unique ON index_coverage_log (page_url, check_date);
CREATE INDEX idx_index_coverage_page ON index_coverage_log (page_url);
CREATE INDEX idx_index_coverage_verdict ON index_coverage_log (verdict);
```

Table 2: seo_content_gaps
```sql
CREATE TABLE IF NOT EXISTS seo_content_gaps (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    query TEXT NOT NULL,
    impressions INTEGER DEFAULT 0,
    clicks INTEGER DEFAULT 0,
    avg_position DECIMAL(5,2),
    priority_score DECIMAL(10,2),   -- impressions * (1 - position/100)
    has_dedicated_page BOOLEAN DEFAULT FALSE,
    suggested_page_type TEXT,       -- 'blog_post', 'service_page', 'location_page'
    status TEXT DEFAULT 'new' CHECK (status IN ('new', 'planned', 'created', 'dismissed')),
    detected_at DATE NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE UNIQUE INDEX idx_content_gaps_query ON seo_content_gaps (query, detected_at);
CREATE INDEX idx_content_gaps_priority ON seo_content_gaps (priority_score DESC);
```

<task type="auto">
  <name>Task 2: Build and deploy TG-96 GSC Daily Sync workflow</name>
  <files>automation/n8n-workflows/TG-96-gsc-daily-sync.json</files>
  <action>
Build TG-96 as the daily GSC data sync workflow. This replaces TG-45's simple weekly pull with a comprehensive daily sync of both page-level and query-level data.

Workflow structure:
```
Schedule Trigger (cron: 0 11 * * * = daily 6 AM CT)
  -> Code Node "Calculate Sync Date"
     // GSC has 2-day data lag
     const syncDate = new Date(Date.now() - 2*24*60*60*1000).toISOString().split('T')[0];
     return [{ json: { syncDate, siteUrl: 'https://tgyardcare.com/' } }];

  -> HTTP Request "Fetch Page Data"
     POST https://www.googleapis.com/webmasters/v3/sites/https%3A%2F%2Ftgyardcare.com/searchAnalytics/query
     Auth: Bearer token from $vars.TG_GOOGLE_SEARCH_CONSOLE_TOKEN (or Google OAuth2 credential if available)
     Body: {
       "startDate": "{{ $json.syncDate }}",
       "endDate": "{{ $json.syncDate }}",
       "dimensions": ["page"],
       "rowLimit": 25000,
       "startRow": 0,
       "dataState": "final"
     }

  -> Code Node "Transform Page Rows"
     // Map GSC response rows to gsc_pages schema
     const syncDate = $('Calculate Sync Date').first().json.syncDate;
     const rows = $json.rows || [];
     return rows.map(r => ({
       json: {
         page: r.keys[0],
         date: syncDate,
         clicks: r.clicks,
         impressions: r.impressions,
         ctr: Math.round(r.ctr * 10000) / 10000,
         avg_position: Math.round(r.position * 100) / 100
       }
     }));

  -> HTTP Request "Upsert gsc_pages"
     POST https://lwtmvzhwekgdxkaisfra.supabase.co/rest/v1/gsc_pages
     Headers:
       apikey: SUPABASE_ANON_KEY_PLACEHOLDER
       Authorization: Bearer SUPABASE_SECRET_KEY_PLACEHOLDER
       Content-Type: application/json
       Prefer: resolution=merge-duplicates,return=minimal
     Body: {{ $json }} (array of page objects)

  -> HTTP Request "Fetch Query Data"
     POST https://www.googleapis.com/webmasters/v3/sites/https%3A%2F%2Ftgyardcare.com/searchAnalytics/query
     Body: {
       "startDate": "{{ syncDate }}",
       "endDate": "{{ syncDate }}",
       "dimensions": ["query"],
       "rowLimit": 25000,
       "startRow": 0,
       "dataState": "final"
     }

  -> Code Node "Transform Query Rows"
     // Map GSC response to gsc_search_queries schema
     // NOTE: When querying with only ["query"] dimension, there's no page column
     // Store page as null or empty -- the query dimension gives aggregate stats across all pages
     const syncDate = $('Calculate Sync Date').first().json.syncDate;
     const rows = $json.rows || [];
     return rows.map(r => ({
       json: {
         query: r.keys[0],
         page: '',  // query-only dimension has no page
         date: syncDate,
         device: 'all',
         country: 'all',
         clicks: r.clicks,
         impressions: r.impressions,
         ctr: Math.round(r.ctr * 10000) / 10000,
         position: Math.round(r.position * 100) / 100
       }
     }));

  -> HTTP Request "Upsert gsc_search_queries"
     POST https://lwtmvzhwekgdxkaisfra.supabase.co/rest/v1/gsc_search_queries
     Headers: same as gsc_pages upsert
     Prefer: resolution=merge-duplicates,return=minimal
     Body: {{ $json }} (array of query objects)

  -> Execute Sub-workflow "Trigger TG-97"
     workflowId: TG-97 (placeholder -- will be set when TG-97 is deployed in Plan 03-03)
     Input: { syncDate: "{{ syncDate }}", pagesProcessed: {{ page_count }}, queriesProcessed: {{ query_count }} }
```

CRITICAL IMPLEMENTATION NOTES:
1. Query page and query dimensions SEPARATELY -- never combine dimensions in one API call (returns aggregated/incomplete data).
2. Always use dataState: "final" to get finalized data only.
3. The gsc_search_queries UNIQUE index is (query, page, date, device). Set page='' and device='all' for query-only syncs.
4. Use resolution=merge-duplicates header for upserts (works because of UNIQUE indexes on both tables).
5. For pagination: TG has ~65 pages and likely <25K queries, so single requests per dimension should suffice. But add a Code node comment noting that if rows.length === 25000, pagination is needed (startRow += 25000).
6. The Execute Sub-workflow for TG-97 should use a placeholder workflow ID. Add a comment noting it needs to be updated after TG-97 is deployed.
7. Use $vars pattern for GSC token: $vars.TG_GOOGLE_SEARCH_CONSOLE_TOKEN. If this doesn't work at runtime, fall back to PLACEHOLDER pattern per Phase 2 convention.
8. Use PLACEHOLDER strings for Supabase keys (SUPABASE_ANON_KEY_PLACEHOLDER, SUPABASE_SECRET_KEY_PLACEHOLDER) -- real values injected at deploy time.
9. fullResponse:true on ALL Supabase GET nodes (Phase 2 lesson).
10. The scheduleTrigger cron should be `0 11 * * *` (6 AM CDT = UTC-5).

Deploy via n8n API. Save JSON locally. Set workflow to ACTIVE.
